# Starter Roadmap

Version: 0.1.0
Last updated: 2026-06-04

## Roadmap Rule

Do not try to move every dimension and spine at once. Pick one valuable workflow, identify the binding constraint, and make the smallest change that moves the workflow one level forward.

If the change is larger than one workflow, use the transformation model to decide whether the next investment is AI working skill, work design depth, reliable AI operations, or organizational adaptability.

## Level 1 To Level 2: From Experiments To Routine Assistance

Goal: make AI safe, visible, and useful for everyday work.

Actions:

- choose a small set of approved AI tools
- define allowed, review-required, and prohibited use cases
- train people on prompting, output review, and data boundaries
- publish 5 to 10 examples tied to real recurring tasks
- create a shared place for useful prompts and lessons

Evidence that the move worked:

- staff use AI weekly for known tasks
- outputs are reviewed before use
- people can name where AI is useful and where it is risky
- managers can see examples of actual AI-assisted work

## Level 2 To Level 3: From Personal Use To Embedded Workflow

Goal: make AI part of the way priority work gets done.

Actions:

- select 2 or 3 workflows with enough volume or value to matter
- document the current workflow and the AI-assisted version
- create reusable prompts, examples, source packs, or context instructions
- define human review points and acceptance criteria
- track simple before-and-after measures

Evidence that the move worked:

- the workflow document includes AI as a standard step
- new team members can follow the pattern
- review quality is consistent
- time, quality, or throughput improves enough to justify keeping the change

## Level 3 To Level 4: From Embedded Workflow To Orchestration

Goal: connect repeated AI-assisted steps into a controlled multi-step workflow.

Actions:

- choose one high-volume or high-friction workflow
- map each step, system, decision point, and handoff
- define which steps AI can execute and which require human approval
- connect only the minimum systems needed for the pilot
- add audit trails, escalation paths, cost controls, and rollback steps
- run the workflow with human oversight before increasing autonomy

Evidence that the move worked:

- the workflow runs across multiple steps without manual re-entry
- approval gates are explicit
- failures are visible and recoverable
- the team can explain cost, quality, and risk performance

## Level 4 To Level 5: From Orchestration To AI-Native Operation

Goal: redesign selected functions around AI-first operation where the value justifies the risk and investment.

Actions:

- identify which functions truly benefit from AI-native design
- define the operating metrics that matter
- manage AI workflows as production systems
- establish regular governance and performance review
- invest in context quality, evaluation, monitoring, and incident response
- redesign roles around direction, exception handling, quality, and improvement

Evidence that the move worked:

- AI-first operation is documented for selected functions
- human-only work is reserved for judgment, exceptions, relationships, or strategy
- the system improves over time based on measured performance
- leadership can explain value, risk, cost, and accountability

## 30-Day Starter Pilot

Use this when a team is at Level 1 or 2 and wants a practical first move.

| Week | Focus | Output |
| --- | --- | --- |
| 1 | Pick one workflow and gather evidence. | Workflow scope, current pain points, baseline measure. |
| 2 | Design the AI-assisted version. | Prompt/context pack, review checklist, usage boundary. |
| 3 | Run the workflow with real work. | 5 to 10 examples, issues log, early value signal. |
| 4 | Decide whether to keep, revise, or stop. | Updated workflow, scorecard, next action. |

## Common Failure Modes

- Buying more tools when the blocker is workflow design.
- Running demos that never become documented work.
- Measuring usage but not quality.
- Automating a process before the manual version is understood.
- Treating governance as a final-stage activity instead of a design input.
- Chasing Level 5 when Level 3 would create most of the value.
